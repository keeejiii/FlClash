# FlClash Fork README（中文）

> 适用于：`keeejiii/FlClash`
>
> 这不是对原版 FlClash 的替代说明，而是这个 fork 相对原版的**差异说明**。

---

## 一、这个 fork 是干什么的

这个 fork 当前的主要方向，不是改 UI 风格，也不是大规模重构架构，而是：

> **在尽量不扩大与上游冲突面的前提下，优化 Android 后台活跃度、降低发热和耗电，并尽量保持后台代理稳定。**

更具体地说，这个 fork 当前遵循的是一条非常明确的运行模型：

- **前台：UI 完整体**
- **后台：UI 静默**
- **日志：无论前后台都完整保留**

简单说：

- **不是做功能最多的分叉**
- **不是做最激进的 Android 特化版**
- **而是做一个更偏稳态、更克制的自用优化版本**

---

## 二、相对原版 FlClash，这个 fork 当前的优势

---

## 1. Android 后台活跃度更低

这是当前最核心的差异。

### 原版特征（按本次 fork 改动前的观察）
原版在 Android 上，后台仍可能持续保留一些对前台 UI 更有意义、但对后台代理本身帮助不大的活跃逻辑，例如：

- traffic / totalTraffic 的刷新链路
- 前台通知的 speed 展示链路
- 前台不可见时仍然存在的部分高频 UI 事件消费
- periodic timer 在后台仍然存在

### 这个 fork 的做法
这个 fork 把后台行为尽量收紧成：

- 后台不再无意义刷新 traffic
- 后台真正停 foreground timer
- 通知栏不再显示 speed
- 后台不再消费高频 UI 事件（delay / request）
- 连接页轮询在后台直接停止，回前台再恢复
- 日志页显示层在后台静默，但日志采集与保留本身完全保留
- 后台保留全部级别日志，用于调试与排障

同时，前台不再做额外的页面级收缩，保持 UI 完整体：

- `traffic` 正常更新
- `delay` 正常更新
- `request` 正常更新

换句话说：

> **前台行为已基本回到接近原版 FlClash 的完整 UI 逻辑；差异主要集中在后台 UI 静默和少量 Android 运行辅助行为。**

### 结果
理论和体感上，这会更有利于：

- 降低待机发热
- 降低后台耗电
- 减少系统将其视为“高活跃后台”的机会

---

## 2. 网络切换后的恢复更主动

原版在 Wi‑Fi / 流量切换场景下，可能存在：

- 旧连接拖尾
- 切网后一段时间才恢复正常
- 后台挂着时切网后短时体验发闷

这个 fork 增加了一个轻量网络变化响应：

- 仅在 Android 主网络类型变化时触发
- debounce 后执行一次 `closeConnections()`
- 不重启 core，不停 VPN，不停 service

### 相对原版的优势
- 切网后旧连接清得更及时
- 恢复速度更主动
- 比直接 stop/start 更温和

---

## 3. 对后台留存更友好，而不是更激进

这个 fork 的省电方向，不是通过“智能暂停代理”换省电，而是：

> **尽可能减少后台无意义工作，但不主动去动 VPN/service 主链。**

也就是说它没有引入：

- smartStop / smartResume
- aggressive restart
- native 复杂状态机
- service 生命周期重构

### 为什么这是优势
如果你的目标是：

- Android 上更省电一点
- 但不想因为激进优化伤到后台代理可用性
- 同时还想继续同步 upstream

那么这种克制路线比“功能很多、状态机很深”的 fork 更适合长期维护。

---

## 4. 与 upstream 的冲突面更小

这个 fork 当前的主要改动，集中在：

- `lib/application.dart`
- `lib/providers/action.dart`
- `lib/manager/app_manager.dart`
- `lib/core/event.dart`
- 少量 Android 通知 / suspend 模块

### 相对原版分叉方案的优势
没有把大量 Android native 行为改造成一套新的运行时体系，所以：

- 后续 cherry-pick / release 同步更容易
- 冲突主要集中在少数文件
- 更容易看清“哪些改动是自己加的”

---

## 三、这个 fork 当前做了哪些关键优化

### 已完成优化

#### 1. 后台不再无意义刷新 traffic / totalTraffic
适用场景：
- App 切后台
- 用户并不在看前台 UI

#### 2. 去掉通知栏 speed 显示
适用场景：
- 不需要通知栏实时监控 speed
- 更重视省电和静态前台通知

#### 3. 后台真正停 foreground timer
适用场景：
- 减少 Dart 层每秒 wakeup

#### 4. 网络变化后主动清理旧连接
适用场景：
- Wi‑Fi ↔ 流量切换
- 网络重连

#### 5. 后台跳过高频 UI 事件
适用场景：
- App 不在前台时
- 降低后台 UI 事件处理成本

#### 6. 连接页轮询纳入后台静默
适用场景：
- App 切后台后不再继续每秒刷新连接页列表
- 回到前台时恢复连接页刷新

#### 7. 日志页显示层后台静默，但日志链路完整保留
适用场景：
- 后台时日志继续采集与保留
- 但日志页面本身不再持续推动 UI 刷新
- 回到前台后再同步显示

#### 8. 补 Doze 状态监听
适用场景：
- 息屏 / Doze
- 挂起判断更完整

#### 9. 日志在前后台都完整保留
适用场景：
- 后台调试
- 问题复现与排障

## 四、这个 fork 的取舍是什么

任何 fork 都不是“白赚”，这个版本也有明确取舍。

### 主动放弃的东西

#### 1. 通知栏实时 speed
如果你很依赖通知栏看速度，这个 fork 不适合你。

#### 2. 激进的 Android 智能运行态控制
没有引入：
- smart stop
- smart resume
- native 网络状态复杂策略

如果你追求的是“像深度 Android 定制代理那样高度智能的运行态”，这个 fork 当前不是那个方向。

---

## 五、这个 fork 适合谁

更适合：

- 主要在 Android 上使用 FlClash
- 体感上对发热和耗电比较敏感
- 希望后台更克制，但不要破坏代理本身稳定性
- 仍然希望后续继续同步原版 FlClash 上游更新

不太适合：

- 非常依赖通知栏实时 speed
- 想要一整套激进 Android 智能状态机
- 想把 fork 做成 BettBox 那种更重度分叉路线

---

## 六、和原版相比，最大的差异一句话总结

### 原版更通用
它更像一个功能完整、通用、多平台一致的主线项目。

### 这个 fork 更偏 Android 后台稳态治理
它不是在追求“功能更多”，而是在追求：

> **前台 UI 完整、后台少做事、日志不断流，同时尽量不伤主运行链。**

---

## 七、如果后续继续维护，这个 fork 的建议方向

如果后续继续沿这个 fork 走，建议保持原则不变：

1. **优先削减后台无意义工作**
2. **保持前台 UI 完整，不轻易做复杂页面级特判**
3. **日志默认全保留，优先服务排障**
4. **尽量不碰 VPN / service 主生命周期**
5. **优先 Dart 层和轻量 Android 辅助模块改动**
6. **在确认收益前，不轻易引入深度 native 状态机**
7. **所有优化都优先考虑后续和 upstream 的冲突成本**
